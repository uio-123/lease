## 第 3 页 · 项目概述

随着住房租赁市场的规范化和线上化趋势，特别是城市青年群体流动性的增强，租赁企业迫切需要数字化系统来提升房源周转效率和客户响应速度。

从国家政策层面看，住房租赁市场发展强调**规范化、透明化和信息化**，鼓励通过数字平台提高住房供给匹配效率、降低线下沟通成本。从地方政策层面看，城市更新、人才住房和保障性租赁住房等业务均要求运营主体具备基础数据管理、服务跟踪和风险控制能力。

用户单位当前面临四个突出问题：一是**房源信息分散**，线下登记或多系统维护容易造成数据不一致；二是**预约流程不稳定**，人工沟通容易遗漏或重复安排；三是**租约管理缺少闭环**，合同和租期状态难以追踪；四是**图片与附件管理成本高**。

本项目选用 Spring Boot 3、Vue3、MySQL、Redis 和 MinIO 技术栈。生命周期模型选择**增量迭代模型**——因为核心业务可以拆分为若干独立增量，每个增量都可以形成可运行版本，便于并行开发和阶段演示。

## 第 4 页 · 项目可行性论证

我们从三个维度分析了项目的可行性：

- **技术可行性**：后端采用 Spring Boot 多模块工程，分为 model、common、web 三层，MyBatis-Plus 降低 CRUD 开发复杂度，JWT 实现登录认证，MinIO 管理图片存储，Redis 承担缓存。前端管理端采用 Vue3 与 Element Plus，租客端采用 Vue3 与 Vant，技术栈均为成熟框架。

- **经济与进度可行性**：技术栈完全开源，不存在许可证成本。小组两人分工推进，组员A侧重后端、数据库、部署与测试，组员B侧重前端、UI、需求文档与汇报材料，在 14 周左右完成全部任务。

- **操作与法律可行性**：项目不涉及商业部署和敏感数据，所有开源框架均遵循 Apache 2.0 或 MIT 协议，不存在版权纠纷。后端通过 Spring Boot 多 Profile 机制支持环境切换，本地基础服务由 Docker Compose 一键启动。

## 第 5 页 · 用户角色与顶层 DFD

系统定义了三种用户角色：

- **租客**：通过移动端完成找房筛选、查看详情、预约看房、查看租约和浏览历史。
- **运营管理员**：维护公寓和房源信息，处理预约和租约。
- **系统管理员**：负责用户、岗位和角色权限的配置管理。

右侧展示的是系统的**顶层数据流图**，展示了系统与租客、管理员、短信服务和对象存储之间的核心交互。系统作为中央处理节点，接收来自不同外部实体的数据输入，处理后返回相应的输出结果。

## 第 8 页 · 功能性与非功能性需求

**功能性需求**方面，系统涵盖五大模块：

1. **认证模块**：手机号验证码登录与自动注册。
2. **房源模块**：多维度搜索筛选、详情查看、图片轮播。
3. **预约模块**：预约提交、状态跟踪、管理员处理。
4. **租约模块**：租约创建、状态管理、列表查看。
5. **后台管理模块**：公寓/房间维护、属性配置、用户与权限管理。

**非功能性需求**方面，我们重点关注：

- **性能**：房源列表接口应在 1 秒内返回，预约提交在 2 秒内完成，支持 200 并发访问。
- **安全**：JWT 登录认证，接口拦截器统一校验，三级权限控制（用户-角色-菜单）。
- **可用性**：前后端分离架构，后端可独立重启，Docker Compose 环境一致。
- **可维护性**：后端按模块划分，职责清晰，便于扩展。

## 第 9 页 · 系统架构图

系统采用**前后端分离架构**，从上到下分为三层：

- **前端层**：后台管理端使用 Element Plus，适合复杂表单与表格操作；租客端使用 Vant H5 组件库，针对移动端优化交互体验。

- **后端层**：基于 Spring Boot 多模块工程结构——model 模块包含实体和枚举，common 模块封装工具类和异常处理，web 模块进一步拆分为管理端接口和租客端接口。

- **基础设施层**：MySQL 负责持久化存储，Redis 承担缓存和验证码存储，MinIO 统一管理图片等静态文件。全部通过 Docker Compose 统一部署。

这种分层架构的好处是职责清晰、独立部署、便于团队并行开发。

**管理后台**包括：公寓管理、房间管理、预约处理、用户管理、系统配置、属性/标签/设施维护、租约管理等运营功能。

**租客端**包括：找房筛选、房源详情、预约看房、我的预约、我的租约、个人中心、浏览历史等面向终端用户的功能。

**基础设施层**为上层提供 MySQL 持久化、Redis 缓存、MinIO 图片存储和 JWT 认证等通用支撑能力。

## 第 11 页 · 数据库设计

数据库以**公寓、房间和租约**作为核心业务链路。核心表包括：

- **apartment_info**（公寓信息）与 **room_info**（房间信息）是房源主数据。
- **room_attr_value**、**room_facility**、**room_label**、**room_payment_type**、**room_lease_term** 等关系表描述房间的各类属性。
- **view_appointment**（预约记录）与 **lease_agreement**（租约合同）承载核心业务流程。
- **system_user**、**system_post**、**system_role**、**system_menu** 承载后台组织架构与权限管理。

这里展示的是系统的 ER 图和核心表结构，完整的数据表清单可以参考报告附录 A。

## 第 12 页 · 租客端 UI 设计

租客端采用 **Vant 移动端组件库**，覆盖完整的用户旅程：

- **首页找房**：支持区域、价格、标签等多维度筛选，以卡片列表形式展示房源。

## 第 13 页 · 管理后台 UI 设计

管理后台采用 **Element Plus 组件库**，采用左侧菜单、顶部标签页、主体表格与抽屉/弹窗表单的组合布局。

核心功能页面包括：

- **公寓管理**：支持新增、修改、上下架操作和图片管理。
- **房间管理**：支持批量维护房间属性、设施、标签和费用。
- **预约管理**：以表格形式展示预约列表，支持状态变更和处理操作。
- **租约管理**：覆盖租约从创建到归档的全生命周期。
- **用户与权限管理**：通过用户-角色-菜单三级体系控制访问范围。

这套界面设计适合运营人员进行日常的重复性管理操作。

## 第 14 页 · 网络设计与数据量分析

系统的**网络架构**分为三层：

- **前端层**：通过 Nginx 或开发服务器承载静态资源，管理端和租客端独立部署。
- **后端层**：Spring Boot 提供 RESTful 接口，管理端运行在 8080 端口，租客端运行在 8081 端口。
- **基础设施层**：MySQL、Redis 和 MinIO 部署在受控网络内，通过 Docker Compose 编排。

**数据量分析**方面：每公寓大约需要 1.5 MB 的图片存储空间；日均预约操作在 200 次以上；接口响应时间控制在 2 秒以内，关键接口通过数据库索引和分页插件保证性能。

## 第 15 页 · 编码实现——后端

后端采用 **Spring Boot 3.0.5 + Java 17**，多模块工程结构：

- **model 模块**：包含实体类、枚举和 DTO。
- **common 模块**：封装统一返回结构 Result、ResultCodeEnum 枚举、全局异常处理器 GlobalExceptionHandler、JWT 工具类、登录用户持有器、MyBatis-Plus 自动填充配置和 MinIO 配置。
- **web 模块**：进一步分为 web-admin（管理端接口，/admin 前缀）和 web-app（租客端接口，/app 前缀）。

技术亮点包括：MyBatis-Plus 的自动填充功能自动管理 create_time 和 update_time；JWT 配合拦截器统一校验用户状态；Knife4j 集成 Swagger 3 提供在线接口文档；MinIO 解耦图片管理与数据库。

---

## 第 16 页 · 编码实现——前端

**管理后台**使用 Vue3 + Element Plus + Vite：

- 实现数据表格展示、表单验证、图片上传、分页筛选等典型管理功能。
- 路由采用静态配置，页面组件按功能模块组织在对应目录下。
- 支持抽屉式编辑，提升操作效率。

**H5 租客端**使用 Vue3 + Vant + Pinia：

- 针对移动端优化首页展示、图片轮播、上拉加载和预约交互。
- 使用底部 TabBar 导航，API 请求统一封装。
- 状态管理通过 Pinia store 模块维护。

前后端通过 HTTP API 通信，使用 JWT Token 进行身份验证。

## 第 17 页 · 测试

测试采用**白盒测试与黑盒测试**相结合的策略：

**白盒测试案例**：使用 JUnit 和 Mockito 覆盖核心 Service 层方法，验证分支覆盖和边界值。测试环境使用本地 Docker 服务，保证数据库和缓存可重复启动。
